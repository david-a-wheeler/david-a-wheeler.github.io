Make Your Open Source Software GPL-Compatible. Or Else.
by David A. Wheeler

This essay is a call to all those developers who are developing open source software / Free Software. Developers: Please, where possible, use an existing widely-used license for your software that is known to be compatible with the General Public License (GPL), such as the GPL, LGPL, MIT/X, or BSD-new licenses. And, if you create your own license - and please try not to - make sure that your new license is compatible with the GPL. You don't need to use the GPL - not even the Free Software Foundation (FSF), the most avid GPL proponent, claims that absolutely all software must be GPL-licensed (e.g., see Richard Stallman's comments on Ogg Vorbis). But make sure that your software's license is GPL-compatible, i.e., that you can combine your code with code released under the GPL into one larger program.

Why? The simple reason is that making your software GPL-compatible ensures that you (and other developers) can reuse other GPL'ed software. Even if you don't like the GPL yourself, many potential co-developers do... and your project is more likely to be successful if you accommodate them. Usually even avid GPL proponents will support software with other GPL-compatible licenses. But if your software isn't GPL-compatible, many OSS/FS developers are likely to abandon your project and create a GPL-compatible version so they can take advantage of GPL'ed code.

And there's a lot of GPL'ed code to take advantage of. Freshmeat.net keeps statistics on licenses, and on March 5, 2002, I found that 65.16% of the 19,882 software branches (packages) followed by Freshmeat are GPL-licensed (the next two most popular were LGPL, 5.45%, and the BSD licenses, 5.18%). The same query examining Sourceforge.net supported projects on March 5, 2002 found that the GPL accounted for 73% of the 22,489 ``open source'' projects hosted on Sourceforge.net (next most popular were the LGPL, 10%, and the BSD licenses, 7%). In my paper More than a Gigabuck: Estimating GNU/Linux's Size, I found that Red Hat Linux, one of the most popular GNU/Linux distributions, had over 30 million physical source lines of code in version 7.1, and that 50.36% of the lines of code were licensed solely under the GPL (the next most common were the MIT license, 8.28%, and the LGPL, 7.64%). If you consider the lines that are dual licensed (licensed under both the GPL and another license), the total lines of code under the GPL accounts for 55.3% of the total. Savannah hosts 693 mostly-GPL projects (as of March 28, 2002) and strongly encourages GPL-compatible licenses. You ignore this amount of code and support at your project's peril.

Several high-profile projects have had to undergo painful changes to make themselves GPL-compatible. The widely-used Python language and text editor vim recently underwent major license changes to become GPL-compatible. Mozilla is also undergoing a complex relicensing scheme, again, to resolve perceived incompatibilities between their older licenses and the GPL. While it's possible to relicense software to make it GPL-compatible, it's quite difficult - you're much better off starting off GPL-compatible in the first place. Thankfully, many projects don't have to endure this transformation. The W3C, for example, explicitly states that all its software releases are OSS/FS and are compatible with the GPL - showing that they believe GPL compatibility is very important.

License analysis is complicated, and best left to lawyers. The FSF developed the GPL and maintains a list of licenses that it believes to be compatible and not compatible with the GPL. My point isn't to debate their legal analysis; I just want to show that determining GPL compatibility can be complicated. Thus, the best approach is to keep things simple.

So, how do you make sure that your software is GPL-compatible? Fortunately, this is easy. One way, of course, is to use the GPL. Another is to use a license that is known to be GPL-compatible, such as the LGPL, MIT/X, and BSD-new (modified BSD) license (but don't use the old/original BSD license, which had many practical problems; Berkeley has switched to the new BSD license for their own software). A third alternative is to dual license the program: release it under more than one license, one of which is the GPL, and let the user pick which one they'll live under (widely-used projects such as Perl and Mozilla do this). If you're submitting changes to an existing project that uses a GPL-incompatible license, include a note in your submission that you're dual-licensing your changes with the GPL; that will make it easier for the project to dual-license the entire program later. In any case, make sure that your OSS/FS software is GPL-compatible, or your project may never have the support you wanted.
