% =========================================================================
% INLINE CODE / IDENTIFIER MARKUP CONVENTION
% =========================================================================
% \code{...} is our markup for a code identifier or single-token snippet,
% e.g. \code{source_corresponds_to_executable}, \code{formulas(goals)}.
% This makes it easy for us to handle this very common case.
% It CANNOT be used in a "moving argument" (a \section/\subsection title,
% a \caption, a \footnote, or the like).
%
% This is a small home-grown verbatim command (NOT url-based): it selects the
% typewriter font and makes  _ ( ) - . /  active so that each prints literally
% (no escaping needed) and permits a line break AFTER it. Long identifiers
% thus wrap cleanly at underscores, hyphens, dots, slashes, and parentheses,
% including after the "(" in formulas(goals), which a url-based command
% refuses to break. Any genuine overflow is reported normally (a url command
% would hide it silently). Do NOT put \allowbreak inside \code;
% the active characters already insert the breaks.
%
% LIMITATION: because \code changes catcodes to read its argument verbatim, it
% CANNOT be used inside a "moving argument" -- a \section/\subsection title, a
% \caption, a \footnote, or the like. There the argument is tokenized before
% \code can change catcodes, so an underscore would raise a "Missing $" error
% if incorrectly used in a moving argument.
% Making \code robust everywhere is complex, and we don't need to do it.
% In those places use plain \texttt{...} instead and escape underscores as \_,
% e.g. \texttt{source\_corresponds\_to\_executable}, with all the other
% instructions involving \texttt{...}.
%
% Use plain \texttt{...} for multi-token snippets that contain spaces,
% e.g. \texttt{stage2 = compile(sA, stage1, ...)}. There, escape underscores
% as \_ and add \allowbreak{} by hand wherever a line break is wanted.
%
\makeatletter
\begingroup
\catcode`\_=\active \catcode`\(=\active \catcode`\)=\active
\catcode`\-=\active \catcode`\.=\active \catcode`\/=\active

\begin{abstract}
Diverse double-compiling (DDC) counters the classic ``trusting trust'' attack by using a second, trusted compiler to verify the compiler under test. It complements reproducible builds, a technique that confirms a binary matches its source but cannot by itself necessarily detect a subverted compiler. DDC has been formally proved correct, and its assumptions have been proved consistent. However, the DDC assumptions have not been shown to be independent. Independence matters in practice: anyone applying DDC to verify a compiler must separately verify that each assumption holds. Proving the assumptions independent shows that none is implied by the others, so none of that verification effort is wasted. This paper demonstrates that the assumptions of all three DDC proofs are independent, constructing a countermodel for each assumption. Along the way, we document subtle pitfalls encountered during automated countermodel generation. These experiences support a broader case: mathematical tools such as provers and model finders should be designed to be maximally safe, actively countering likely mistakes through defensive engineering and language affordances rather than silently accepting misinterpreted proof requests.
\end{abstract}

\keywords{diverse double-compiling, DDC, trusting trust, supply chain security, open source software, formal proof, assumption independence, tool safety, defensive tool design, maximally safe tools}

\section{Introduction}

Most software security issues have, in principle, well-un\-der\-stood solutions:

\begin{enumerate}
	\item Many programs contain unintentional vulnerabilities. These vulnerabilities can in principle be identified using human review, traditional static and dynamic analysis, and artificial intelligence (AI).
	\item Programs may have intentional vulnerabilities in their source code. These can often be detected through similar methods, in particular, by reviewing their source code. Attackers can create ``underhanded'' source code written to try to mislead reviewers, but there are countermeasures against underhanded code that reduce that risk~\cite{wheeler2020underhanded}.
	\item A compiled executable of an ordinary program may not correspond directly to the source code reviewed by others due to causes such as manipulated source releases, subverted builds, subverted distribution, or corruption after loading. Examples include the attack on SolarWinds Orion in 2020~\cite{wheeler2021preventingsolarwinds} and the backdoor in XZ Utils revealed in 2024~\cite{akamai2024xzutils}. Countermeasures exist, including protecting builds (see OpenSSF SLSA~\cite{openssfslsa}), digital signatures (see OpenSSF Sigstore~\cite{openssfsigstore}), and reproducible builds~\cite{reproduciblebuilds}.
\end{enumerate}

Addressing these challenges is not necessarily \emph{easy}, but at least there \emph{are} known ways to do so. Unfortunately, a different and clever attack subverts all of these countermeasures.

\subsection{Trusting Trust Attack}

An Air Force evaluation of Multics by Karger and Schell \cite{karger1974multics,karger2002thirty}, and Ken Thompson's Turing Award lecture (``Reflections on Trusting Trust'') \cite{thompson1984trust}, showed that compilers can be subverted to insert Trojan horses into critical software, including themselves, as part of the compilation process. If this ``trusting trust'' attack goes undetected, even complete analysis of a system's source code will not find the malicious code that is running. Re-running the compiler does no good, because such a compiler can self-subvert. The ability to subvert future versions of tools makes this attack challenging to counter.

For decades, the industry treated the ``trusting trust'' attack as an unresolvable problem. Yet if you cannot completely trust the compiler executable used to build your tools, you cannot trust anything built with it, including other compiler executables descended from its compilation. This is especially important for open source software, where people verify source code but execute compiled executables.

\subsection{Countering the Trusting Trust Attack}

Diverse double-compiling (DDC) counters the trusting trust attack by using a second, trusted compiler to compile the parent of the compiler under test, then using that result to compile the source code of the compiler under test \cite{wheeler2010ddc}. If the output of this process matches the original compiler executable, then the executable corresponds to its source code. DDC does not require that arbitrary different compilers produce the same executable output given the same input. Indeed, this would be extremely unlikely in the real world. DDC instead requires determinism by the parent compiler of the compiler under test. It is generally desirable that a compiler be deterministic, and many real-world compilers are, or can be made so via a flag.

Another countermeasure against the trusting trust attack is to implement a bootstrappable build~\cite{bootstrappableorg}. Bootstrappable builds minimize the number and size of bootstrap binaries, which are then used to build the next layer, which in turn builds the next, and so on until everything else is built. This is a promising approach, but challenging, and many environments are not based on bootstrappable builds. DDC and bootstrappable builds are strongly complementary approaches. Bootstrappable builds limit opaque binaries and address the question of how to get a trustworthy trusted compiler. DDC can use those resulting bootstrapped builds to verify other results that were \emph{not} created with bootstrappable builds.

DDC has been formally (mathematically) proved to produce this result when a set of reasonable assumptions is true \cite{wheeler2010ddc}. The assumptions and goals were defined in first-order logic (FOL), and proofs were generated with Prover9 \cite{mccune2009prover9}. There is strong confidence that these proofs are correct; they were automatically generated, checked step-by-step by a different formally-proven tool, and reviewed by multiple people. The same source also proved that these assumptions were consistent (that is, they did not contradict each other).

The proofs use a systematic notation for the compilers, source code, and intermediate results involved, following the DDC formalization~\cite{wheeler2010ddc}. A symbol's leading letter gives the kind of object (\code{c}, compiler executable; \code{s}, source code; \code{l}, source language; \code{e}, environment) and its suffix gives the role (\code{A}, the compiler under test; \code{P}, its parent; \code{T}, trusted). Table~\ref{tab:notation} summarizes the symbols used throughout this paper.

In fact, there are three proofs involving DDC:

\begin{enumerate}
	\item Goal \code{source_corresponds_to_executable} (5 assumptions): This is the key proof for DDC. It shows that given certain assumptions, if \code{stage2} (the result of the DDC process) and \code{cA} (the original compiler-under-test) are equal, then the executable \code{cA} and the source code \code{sA} exactly correspond. (Filename prefix \code{ddc}.)
	\item Goal \code{always_equal} (9 assumptions): This proves that, under ``normal conditions'' (such as when compiler executables have not been rigged and thus do correspond to their respective source code), \code{cA} and \code{stage2} are always equal. Thus, the first proof is useful, because its assumptions will often hold. This also implies that if \code{cA} and \code{stage2} are not equal, then at least one of its assumptions is not true. (Filename prefix \code{all-repeats}.)
	\item Goal \code{cP_corresponds_to_sP} (3 assumptions): The previous ``\code{always_equal}'' proof does not require that a ``grandparent'' compiler exist, but having one is a common circumstance. This third proof shows that if there is a grandparent compiler, one of the assumptions of proof \#2 can be proved given other assumptions that may be easier to verify (potentially making DDC even easier to apply in this common case). (Filename prefix \code{cP-corresponds-sP}.)
\end{enumerate}

This paper makes two contributions. First, we prove that the assumptions of all three DDC proofs are independent, so none of them is redundant. Anyone applying DDC to verify a compiler must check each assumption separately, but no effort spent doing so is wasted. Second, we make a broader case that mathematical tools such as provers and model finders should be designed to be \emph{maximally safe}: actively alerting on likely mistakes, protecting users from them, and making correct use easy. The two contributions are connected: we obtained the independence result with confidence only because we recognized and overcame subtle ways these tools can silently misinterpret a proof request, so the formal result and the case for safer tools are two outcomes of the same careful process.

\section{Related Work}

The trusting trust attack remains a recognized threat to software supply chain integrity. Reproducible builds confirm that a binary corresponds to its source, but they assume a trustworthy compiler and so do not by themselves counter a subverted compiler~\cite{lamb2022reproducible}. Cox has since reconstructed and annotated Thompson's original self-re\-pro\-duc\-ing compiler, making the attack concrete for a modern audience~\cite{cox2023nih}.

Diverse double-compiling was briefly suggested by Henry Spencer and then developed, named, and demonstrated by Wheeler~\cite{wheeler2005ddc}. Wheeler later generalized DDC to support arbitrary parent compilers and gave machine-checked proofs of its correctness and of the consistency of its assumptions~\cite{wheeler2010ddc}. Skrimstad implemented a self-rep\-li\-cat\-ing attack against the Go compiler and proposed applying DDC with more than one trusted compiler to raise assurance beyond applying DDC only once~\cite{skrimstad2018ddc}. Rosencrantz applied DDC to harden cryptocurrency software against supply-chain attacks~\cite{rosencrantz2023ddc}. Christensen integrated DDC into a CI/CD pipeline so that compiler verification runs automatically as part of the build workflow~\cite{christensen2025ddc}. More broadly, the re\-pro\-duc\-ible-builds community has rebuilt bootstrap compilers such as GNU Mes bit-for-bit across independent distributions and toolchains, which could be used as a trusted compiler for applying DDC~\cite{mesbootstrap2019}. None of these papers proved that the DDC assumptions are \emph{independent}.

\section{Assumption Independence in First-Order Logic}

If a proof uses a set of assumptions, that does \emph{not} mean that the assumptions are independent. It is possible that some of its assumptions can be derived from some of the other assumptions. This is not desirable. When applying DDC in practice it takes effort to gain strong confidence that each assumption is true. It would be better if we could either (1) prove that some assumptions are derived from others (in which case they no longer need to be separate assumptions), or (2) prove that all the assumptions are independent of each other in their respective proofs.

Given some set of consistent assumptions $Q$, and another assumption $P$, then $P$ is independent of $Q$ if both $Q \cup \{P\}$ and $Q \cup \{\neg P\}$ yield consistent assumptions. This is noted, for example, by Kunen~\cite[page xi]{kunen1980set}. It is already formally proved that our entire base set of DDC assumptions is consistent, so the first half of that definition ($Q \cup \{P\}$) is already proved. Therefore, to prove $P$'s independence, we only have to show that $Q \cup \{\neg P\}$ is consistent.

In first-order logic, the easiest way to show that a set of assumptions is consistent is to find a ``model'' where they are all true. Formulas in first-order logic contain abstract syntax symbols (variables, relations, and functions) with no inherent mathematical meaning. A \emph{structure} (or \emph{interpretation}) defines a specific (``concrete'') domain of objects (aka a universe of discourse) and maps these abstract symbols to actual relations, functions, and constants on that domain. An \emph{assignment} (or \emph{valuation}) then maps the free variables to specific elements within that domain. Under this interpretation and assignment, we evaluate the truth value of the formulas. A structure is a \emph{model} of a set of formulas if we can assign concrete values from the domain to the variables such that every formula in the set evaluates to true~\cite{enderton2001logic}.

A model is not a claim about absolute, universal truth across all possible universes. Rather, it is an assertion of \emph{satisfiability}. It is a concrete, mathematically consistent instance (a specific ``world'' with specific assignments of values) where the assumptions are satisfied. To prove independence, we do not need to show that our assumptions are universally true; we just need to show that there exists at least one such satisfying assignment of values (a model) where $P$ is false while all other assumptions in $Q$ remain true.

Hesseling explains that Heyting~\cite{heyting1930regeln} uses this exact approach to prove the independence of his formalization of intuitionistic logic, and credits Bernays with the technique in which ``what we would nowadays call models in which each time ten of the eleven given axioms are true, the eleventh one false\dots\ [by] varying the eleventh axiom and presenting a different model to which the new combination of axioms applies, he proves that all the axioms are independent of each other''~\cite[page 278]{hesseling2003gnomes}.

\section{Our Approach for Finding Models}

To formally establish the independence of the DDC assumptions, we used automated logic tools. Specifically, to show that a set of $n$ assumptions is independent for some proof, we must prove that no single assumption can be derived from the remaining $n-1$ assumptions. We can do that by demonstrating a counterexample.

In our case, for each assumption $a_i$, we must show that there exists a mathematically consistent world where $a_i$ is false, yet all other assumptions in a given proof remain true. If we do this for each assumption for all three proofs, we have shown that all of the assumptions are independent of each other. More precisely, each assumption is independent of all the others taken together (their conjunction), which is strictly stronger than pairwise independence.

We fed our formal assumptions into Mace4~\cite{mccune2009prover9}, which searches for finite mathematical models. Mace4 is not the most cutting-edge model finder today. However, it is still a fine tool, and Mace4 takes the same syntax as Prover9, which was used to generate the original proofs.\footnote{A maintained modernization, LADR-2026 by Machado and Lesyna, has since been released. It builds on the original code, and preserves McCune's algorithms and input syntax, while repairing and extending the tools (for example, native TPTP/TSTP support)~\cite{machado2026ladr}. We deliberately used the same version as in the original DDC proofs, to avoid any behavioral differences. However, this actively maintained version is a natural choice for new work.} Using the tool specifically designed to accept that syntax avoided the risk of mistranslating the statements if we switched to a different system.

\subsection{The Mace4 Variable Scope Trap}

When attempting to formally prove independence using Mace4, early drafts ran into a subtle roadblock while analyzing the second assumption of goal \code{always_equal}. This occurred with the assumption defining a portable and deterministic program, labeled \code{define_portable_and_deterministic}.

Consider this attempted negation of the assumption, where we take the original assertion and wrap it with \texttt{- ( ... )}:

\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={An attempted negation that wraps the assumption in \texttt{- ( ... )}, which Mace4 misinterprets.}]
formulas(assumptions).
  - ( ( portable_and_deterministic(Source,
          Language, Input) &
        exactly_correspond(Executable1, Source,
          Language, Environment1) &
        exactly_correspond(Executable2, Source,
          Language, Environment2)) ->
          ( converttext(run(Executable1, Input,
              EnvEffects1, Environment1),
              Environment1, Target) =
            converttext(run(Executable2, Input,
              EnvEffects2, Environment2),
              Environment2, Target)))
    # label(define_portable_and_deterministic).
end_of_list.
\end{lstlisting}
\end{minipage}

At first glance, this looks reasonable. To negate the rule, we put a negation symbol ($-$) in front and parenthesized the rest. Unfortunately, this causes a catastrophic misinterpretation due to how Mace4 handles unquantified variables.

In Prover9 and Mace4 set to Prolog-style variables, when you use capitalized terms (like \code{Source} or \code{Input}) without an explicit quantifier, the tool treats them as free variables. To process the file, the engine binds these variables by automatically wrapping the entire expression in an implicit universal quantifier ($\forall$). This was appropriate for the original assumption. However, because in this modified assertion the negation symbol ($-$) is written \emph{inside} the statement, Mace4 applied the implicit universal quantifier \emph{outside} the negation. Mathematically, Mace4 interpreted the assumption as:
\[\forall \vec{x} \, \neg (\text{Premise}(\vec{x}) \rightarrow \text{Conclusion}(\vec{x})).\]

For an implication ($\rightarrow$) to be false, the premise must be true \emph{and} the conclusion must be false. Because of the mis-applied universal quantifier, the resulting request was to satisfy an absurd set of requirements.

To prove independence, we simply want to prove that there is \emph{at least one} counterexample. Mathematically, a true counterexample requires an existential quantifier ($\exists$) wrapped around the negation $\exists \vec{x} \, (\text{Premise}(\vec{x}) \land \neg \text{Conclusion}(\vec{x}))$. In raw Mace4 syntax you can manually write out every single variable to force this behavior:

\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={Tediously forcing the intended existential by naming every variable by hand.}]
exists Source Language Input Executable1
       Environment1 Executable2 Environment2
  EnvEffects1 EnvEffects2 Target (
    portable_and_deterministic(Source, Language,
      Input) &
    exactly_correspond(Executable1, Source,
      Language, Environment1) &
    exactly_correspond(Executable2, Source,
      Language, Environment2) &
    - ( converttext(run(Executable1, Input,
          EnvEffects1, Environment1),
          Environment1, Target) =
        converttext(run(Executable2, Input,
          EnvEffects2, Environment2),
          Environment2, Target))
).
\end{lstlisting}
\end{minipage}

Writing this out manually is tedious and highly susceptible to typos or missed variables.

We then realized there was a much safer en\-gi\-neer\-ing-focused solution: let the tool do the negation for us, including handling the quantifiers. Instead of manually negating the assumption, we can move the assumption into a new \code{formulas(goals)} block:

\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={The safe alternative: move the target assumption into a \texttt{formulas(goals)} block.}]
formulas(assumptions).
    % All other DDC assumptions unmodified
end_of_list.

formulas(goals).
    % The assumption to be shown independent
end_of_list.
\end{lstlisting}
\end{minipage}

When Prover9 or Mace4 sees a statement in the \code{goals} block, it uses a built-in automated mechanism that correctly negates the expression behind the scenes. By using the \code{goals} block, we eliminate the risk of quantifier mismatch. Mace4 will search for a model that satisfies all our assumptions while breaking the goal in exactly \emph{one} valid place. If it finds that model, our independence proof is correct, our syntax remains clean, and we avoid subtle scoping traps.

The \code{goals} block is an example of a \emph{syntactic affordance}, that is, a syntactic construct designed to make common constructs easier to \emph{correctly} express. The term \code{goals} is misleading; it stems from the fact that Prover9 is a ref\-u\-ta\-tion-based theorem prover. It would be nicer if \code{goals} had a synonym like \code{negated} when you are not really expressing a goal. Still, the mechanism is a significant aid.

However, using this approach to avoid a trap risks triggering \emph{another} trap.

\subsection{The Mace4 Missing Ending Trap}

Typically, goals follow assumptions in these files, using the
\code{formulas(goals)}
header block to open the section. But what happens in Mace4 if the assumptions list is not terminated with an explicit \code{end_of_list} separator?

The answer, unfortunately, is that this is a major hazard. This tiny syntactic omission can silently change the entire meaning of the proof request without throwing a parser error or even a warning. This behavior creates a dangerous ``gulf of evaluation'' where a human reader interprets the input file one way, but the tools interpret it in a different way.

Consider a typical Prover9 or Mace4 input file where a user defines a set of background assumptions and a target goal to test for independence:

% Use minipage to force this listing to stay in 1 column (skipping to
% the next column if it must). It's short, and splitting the listing
% across columns makes it ugly.
\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={A problematic LADR input block containing a missing boundary separator.}]
formulas(assumptions).
  assumption_1.
  assumption_2.
  % Missing 'end_of_list.'

formulas(goals).
  target_goal.
end_of_list.
\end{lstlisting}
\end{minipage}

To a human eye, the block boundary is obvious. We see \code{formulas(goals)} and naturally assume the tool transitions from parsing assumptions to parsing goals. Many systems do work this way, for example, an HTML processor that reads a list item start (\code{<li>}) will close the previous list item. In the LADR parser used by Mace4 and Prover9, however, \textbf{no such transition occurs}.

The silent semantic shifting of formulas occurs due to specific implementation properties of the sequential engine:

\begin{enumerate}
    \item \textbf{Sequential Term Parsing:} The LADR parser is a sequential engine. When it enters a \code{formulas(list_name).} block, it continuously parses incoming lines as first-order terms (formulas) until it hits the specific, isolated atomic symbol \code{end_of_list}.
    \item \textbf{Functor Misinterpretation:} Because \code{"formulas"} is not a reserved keyword in the lexer, the line \code{formulas(goals)} is parsed as a syntactically valid \emph{compound term} (specifically, a functor of arity 1 with the constant argument \code{"goals"}).
    \item \textbf{Silent Appending:} Because there was no \code{end_of_list} to break the parsing loop, the parser simply appends the compound formula \code{formulas(goals)} to the active \code{assumptions} list.
    \item \textbf{Goal Hijacking:} Every subsequent line, \emph{including} the intended \code{target_goal}, continues to be parsed as an assumption. The parser only exits the assumption list when it finally reaches the single \code{end_of_list} at the end of the file.
\end{enumerate}

This behavior leads to a silent, compounding failure that misleads both the tool and the investigator. In our case, the tool immediately found a model that solved the misinterpreted problem. The investigator, seeing a model generated, falsely concludes \emph{``Excellent, Mace4 found a valid countermodel, so my assumption is independent!''} In reality, the tool proved nothing useful when \code{end_of_list} was missing.

\section{Results}

Once these problems were addressed, Mace4 successfully identified models for all 17 ($5+9+3$) situations. Each model is a countermodel showing a way to satisfy the negation of the target assumption while also satisfying all other background assumptions within its respective proof.

This formally proves that \emph{none} of the DDC assumptions are redundant within their proofs. All 17 models found have a domain size of 2. This small size is not a theoretical constraint, but it does suggest it should be relatively easy for many other automated model finders to find at least one relevant model.

\subsection{Discussion}

Each model, except one, was found in less than a second. The exception was \code{all-repeats-independence-7}, which required 42 seconds of search time. Search time is not important compared to the mathematical confirmation of satisfiability, but this demonstrates the computational feasibility of the check.

There is a good reason that finding the model for \code{all-repeats-independence-7} took longer. This model required negating the assumption \code{definition_cA} which states \texttt{cA = compile(sA, cP, eAeffects, eA, eArun)}. Mace4 searches for a model by assigning values to the cells of each function and relation table over the finite domain. This assumption defines the \code{compile} function as a deeply nested mapping:
\texttt{compile(S, C,\allowbreak{} E,\allowbreak{} R, T) =\allowbreak{} extract(\allowbreak{}converttext(\allowbreak{}run(C,\allowbreak{} retarget(S, T),\allowbreak{} E, R),\allowbreak{} R, T))}.
This definition involves a 5-ary function (\code{compile}), a 4-ary function (\code{run}), and a 3-ary function (\code{converttext}). With our small domain size of $D = 2$, the 3-ary function is not too bad; it only has $2^{(2^3)} = 2^{8} = 256$ possible configurations. However, a 5-ary function has $2^{(2^5)} = 2^{32} = 4{,}294{,}967{,}296$ possible configurations, and a 4-ary function has $2^{16} = 65{,}536$ combinations.

When Mace4 negates the goal (\code{cA} $\neq$ \code{compile(...)}), it must satisfy all the other relevant assumptions. Because the other assumptions contain specific instances of \code{compile} (such as \code{stage1} and \code{stage2}), Mace4 must construct a consistent mathematical universe across this massive combinatoric space of 5-ary and 4-ary constraints (as well as the 3-ary constraint). The 42-second search time reflects the size of this constraint space (the shared \code{compile} table that every assumption in this proof constrains), not any error or special depth in this particular independence result.

Independence here takes two forms, distinguished by a simple syntactic test: whether the negated assumption rests on a \emph{free} symbol or only on \emph{shared} ones. In 11 of the 17 cases the assumption contains a symbol (a constant, function, or relation) that occurs in no other assumption of its proof; examples include the constants \code{stage2}, \code{cA}, and \code{e1effects}, and the \code{extract} function. A countermodel can then falsify that assumption by interpreting its free symbol however it likes, so the independence is immediate. In the remaining 6 cases every symbol is shared with another assumption, so no such one-symbol argument applies. This syntactic split does not track search difficulty. The single model that required a long search was the 42-second \code{definition_cA} case above, and it negates one of the 11 free-symbol assumptions; its cost comes from the shared background theory, not from any depth in that assumption's independence. Either way, no assumption is derivable from the others. Because independence concerns derivability rather than ease of checking, the triviality of an assumption's independence says nothing about how hard that assumption is to verify for a real compiler; each must still be checked on its own when applying DDC.

We could find models using \emph{other} tools, such as Z3 or cvc5, but that is not necessary: one model suffices. We primarily chose Mace4 to avoid translation concerns, but there were other good reasons. Modern SMT solvers like Z3 and cvc5 are optimized to prove unsatisfiability, so they can struggle when searching for satisfying models with first-order quantifiers~\cite{demoura2011smt,reynolds2013fmf}. Mace4 bypasses this quan\-ti\-fi\-er-reasoning difficulty entirely by systematically searching small, finite domains starting at size 2, turning a computationally unbounded search into a simple, highly solvable constraint problem. That said, we expect other tools \emph{could} find models, and that might be interesting future work.

Since the assumptions are independent, applying DDC offers no shortcut: to rely on the security guarantees of any proof in~\cite{wheeler2010ddc}, you must establish every assumption of the proofs you depend on.

\subsection{Limitations}

Our results have a few natural limitations. First, independence is established relative to the specific first-order formalization of DDC as published in~\cite{wheeler2010ddc}. A different proof or set of assumptions for the process might be different. Second, establishing independence requires only a single countermodel per assumption, so we do not claim that the particular models we found are canonical, minimal, or unique; they serve only as witnesses that the required countermodels exist. Third, independence is a statement about derivability alone. It confirms that each assumption must be verified separately, but it says nothing about how difficult any given assumption is to establish for a real compiler. An assumption that is trivially independent may still be the hardest to verify in practice.

\section{Lessons Learned}
\label{sec:lessons}

These experiences suggest several broader lessons.

\subsection{Mathematical Tools Should Be Maximally Safe}

Modern cars do not just go somewhere. They are designed to alert users about unusual events, to attempt to protect people if a mistake is made, and to make it easy to perform typical tasks in a safe way. Modern cars include engine lights, turn and brake signals, seat belts, air bags, bumpers, speedometers, and \emph{many} other features to alert, protect, and simplify correct use. We cannot prevent all collisions, but we can reduce their likelihood and expected impact. Cars that fail to do this are \emph{unsafe at any speed}~\cite{nader1965unsafe}.

Today, cars must include many safety features to be legally sold in many countries~\cite{fmvss208,eugsr2019,unr16}.

Similarly, mathematical tools should be designed to alert on unusual events, attempt to protect their users from mistakes, and simplify correct use. Both humans and AI make mistakes, often following patterns that are easy to predict. There are also common uses that, if made easy, are more likely to be done correctly. A tool need not have \emph{many} such mechanisms, but a good one typically will. Tool makers may omit them from an ill-conceived and narrow conception of the tool's requirements, dismissing such safeguards as irrelevant extras, \emph{exactly} as early car makers once dismissed seat belts and the like. We cannot prevent all mistakes with these tools, but \emph{can} reduce their likelihood and expected impact, while also making them easier to safely use.

Such failures carry a real cost. An attempt at this very result was abandoned over a decade ago when misstatements the tool should have reported kept it from producing the needed models. Corrected, the statements yield models in seconds. That an experienced investigator was stymied for so long indicts the tool itself.

An old proverb says, ``A bad workman always blames his tools''. The proverb is true but often misunderstood; a wise worker takes responsibility for the result, \emph{including} avoiding or fixing bad tools.

\subsubsection{Precedents in Formal Verification}

The idea that a verification tool should actively flag likely mistakes is not new, but it has been pursued piecemeal. In model checking it is institutionalized as \emph{sanity checking}. Beer et al.\ observed that ``a valid formula may hide real problems in the model,'' reported that ``typically 20\% of formulas are found to be trivially valid, and that trivial validity always points to a real problem,'' and concluded that ``[i]f the validity of a formula is trivial, this must be indicated to the user''~\cite{beer2001vacuity}. Kupferman surveys a whole family of such checks, whose ``goal \dots\ is to detect such errors by further automatic reasoning''~\cite{kupferman2006sanity}. Proof assistants sometimes offer analogous affordances: Isabelle's Nitpick is ``a counterexample generator for Isabelle/HOL that builds on Kodkod, a SAT-based first-order relational model finder''~\cite{blanchette2010nitpick}, and its Quickcheck ``uncovers faulty specifications and invalid conjectures''~\cite{bulwahn2012quickcheck}, so that ``[u]sers can identify errors \dots\ by immediate counterexamples rather than by time-consuming unsuccessful proof attempts.'' Each of these, however, is a specific safeguard against a specific mistake.

Our argument is broader: safeguards should be pursued deliberately and pervasively as a standing design goal for provers, model finders, and the underlying logical notation, not added as a special case. Professional and standards bodies such as the ACM and IEEE could help by defining freely available minimum safety requirements for such tools and notations, and requiring that a warning be posted whenever an automated tool is used if it fails to meet minimum safety requirements.

\subsubsection{Examples from This Experience}

There is more than one way to address the variable scope trap. The system could require \emph{all} variables to be bound, for example. Another solution would be to alert on suspicious patterns like a leading \code{-} of a formula with an unbound variable. Once the problem is detected, Mace4 does have the affordance \code{formulas(goals)} to reduce the risk of incorrect negation. That is a welcome capability, though alternative clear names would be an improvement.

Similarly, there is more than one way to deal with a missing ending. The tool could reject a statement starting a new section when a section is already open (this is the approach we prefer). The tool could also quietly close any section in progress when it is asked to start a new section. The point is that the \emph{current} behavior is dangerous.

These are not the only footguns in Prover9 and Mace4. For example, Prover9 treats numerals as ordinary FOL constants, so \code{0 = 1} is \emph{possibly true}, whereas Mace4 assumes different numerals denote different objects, so \code{0 = 1} is never true (unsatisfiable)~\cite{mccune2009mace4input}. It is easy to forget to state that \code{0} and \code{1} are not equal, and Prover9 will then happily let them be the \emph{same} number, silently breaking a proof. The two tools, though meant to accept the same inputs, genuinely disagree about it.

\subsubsection{Similar Tools}

The problem of failing to guard against likely mistakes is not specific to Mace4. It also applies to many tools that support the SMT-LIB format, such as Z3 and cvc5. As a simple example, consider division by zero.

Under the SMT-LIB standard, division is a total function, so $x / 0$ has a value with no constraint on it~\cite{BarFT-RR-25}. A solver may then choose that value to satisfy the other constraints, producing a misleading ``model''. We verified this with Z3 4.16.0 and cvc5 1.3.4: on a problem solvable only by dividing by zero, both reported \code{sat} (satisfiable) with a zero denominator (in effect choosing $0 / 0 = 10$) and no warning, whereas a single ``divisor must be non-zero'' guard made both correctly report \code{unsat} (unsatisfiable). Mace4 makes the opposite trade: rather than assign $x / 0$ a value, it leaves division by zero unevaluated, so it is sound but may be incomplete, and its manual notes it ``might say that there are no solutions when there are valid solutions''~\cite{mccune2009mace4arithmetic}. On the same problem, Mace4 2009-11A does not fabricate a value; it reports no model rather than an unsound one. Its risk is a silently missed solution, not a misleading one, and even that misleads only if a failed model search is mistaken for a proof, which Mace4 is not intended to provide.

Different as these failures are, the two share flawed division-by-zero handling. Neither prevents division-by-zero problems, and neither warns that division by zero shaped a specific result. That silence points to a hierarchy of defaults. Safest is to prevent the mistake automatically (such as by rejecting unconstrained division by zero or inserting the precondition), while allowing an explicit override for the rare case that wants it. Warning by default is acceptable when prevention is not technically feasible. A warning the user must first turn on is far weaker and often inappropriate, but at least it is something. Worst is dangerous by default with no safety at all, which is where the unguarded SMT-LIB behavior sits: it neither prevents the misleading model nor warns about it, and in the versions we tested offered no warning to enable. Simply documenting a problem is inadequate for real-world tool use. These tools are \emph{excellent} if your goal is to get useful answers for perfect users! The \emph{problem} is the lack of perfect users. The reluctance of many tool makers to guard against easily foreseeable mistakes is eerily similar to that of many 1950s car makers.

Division by zero is not the only silent surprise. Fixed-width bit-vectors carry their own, because the same bits mean different things to different operators. The byte \code{\#xFF} is $255$ to the unsigned \code{bvult} but $-1$ to the signed \code{bvslt}, so choosing the wrong comparison silently flips the result, and arithmetic wraps without warning, so \code{\#xFF + \#x01} is \code{\#x00} with no overflow signal~\cite{BarFT-RR-25}. The solver reports no error; the wrong operator or an unnoticed overflow simply yields a confident, wrong answer.

\subsubsection{Exists with Implies}

We can even find risky constructs in the underlying first-order logic notation that many systems are built on. In some ways, these are the most dangerous and need the most emphasis. \emph{All} tools that build on this widely-used notation, and do not take steps to deal with them, can lead to the same problems.

A common translation error involves pairing the existential quantifier (``exists'') with material implication (``implies''), like this:
\[ \exists x (P(x) \rightarrow Q(x)). \]
This pattern is almost always a catastrophic misstatement of user intent. Typically, an investigator intends to assert that there exists some object $x$ which possesses property $P$ and satisfies consequence $Q$, which should be formally stated with a conjunction instead:
\[ \exists x (P(x) \land Q(x)). \]
The previous form is dangerous because $P(x) \rightarrow Q(x)$ is equivalent to $\neg P(x) \lor Q(x)$: any object that is not a $P$ satisfies it vacuously, so a solver can ``confirm'' the claim by pointing at such an object, never exhibiting one that actually has property $P$. The existential quantifier pairs naturally with conjunction, whereas the universal quantifier pairs with implication~\cite{barwise2011lpl}.

The SPARK system has an option that warns about constructs like this because they are such common mistakes~\cite{wheeler2026allsome}.

\subsubsection{Combining For-all and Exists: Allsome}

In standard first-order logic, universal statements of the form $\forall x (P(x) \rightarrow Q(x))$ are susceptible to the trap of \emph{vacuous truth}. If no object in the universe satisfies the antecedent $P(x)$, the entire statement is always true. Put plainly, both ``all Martians are green'' and ``all Martians are not green'' are true if there are no Martians. In security verification, this can be disastrous.

To prevent this error, an investigator must manually chain an existential prerequisite (``a Martian exists''):
\[\bigl(\forall x \, (P(x) \rightarrow Q(x))\bigr) \land \bigl(\exists x \, P(x)\bigr).\]
This is mathematically sound, but repeatedly writing this lengthy compound expression introduces a heavy unnecessary cognitive load. It also invites typos, variable mismatch, or scoping errors.

The ``allsome'' quantifier, written $\forall\exists$ or $\allsome$, is a helpful affordance in such cases. A language that supports this quantifier simplifies guaranteeing that every relevant instance satisfies the given criteria (\emph{all}) and that at least one relevant instance exists (\emph{some}), by providing a quantifier that does both:
\[\allsome x \, (P(x) \rightarrow Q(x)) \;\equiv\; \bigl(\forall x \, (P(x) \rightarrow Q(x))\bigr) \land \bigl(\exists x \, P(x)\bigr).\]
A bounded variant ranges over a class or set $A$:
\[\begin{aligned}
\allsome x \in A \; (P(x) \rightarrow Q(x)) \equiv{} & \bigl(\forall x \in A \; (P(x) \rightarrow Q(x))\bigr) \\
& {}\land \bigl(\exists x \in A \; P(x)\bigr).
\end{aligned}\]
Crucially, the ``some'' requires a member of $A$ that also satisfies the antecedent $P$, not merely that $A$ is nonempty; a witness lying in $A$ yet failing $P$ would leave the implication vacuously true. When $A$ is the entire domain, this form reduces to the first. This defensive language feature eliminates the risk of vacuous truth while keeping the proof source clean, readable, and less error-prone~\cite{wheeler2026allsome}. The ``allsome'' quantifier has been defined, and its key properties proved, in the Metamath Proof Explorer (the \code{set.mm} database)~\cite{metamathmpe}. It would be good to see it in more tools.

\subsection{Use AI to Review}

Both of the Mace4 problems listed above were detected by AI, specifically Google Gemini. Mace4 and Prover9, along with their documentation, have been widely available for years. Many AI systems already know about these tools and their foibles.

We think it is important today to have at least one AI review proofs we rely on in the real world, in addition to human review. While we discuss only one set of proofs, this demonstrates that AI \emph{can} sometimes find important problems that human reviewers might miss. This must be kept in context: AI also generates false claims, expresses confidence in false statements, and offers no guarantee of finding every problem. Depending \emph{only} on AI review would be a mistake. AI review should \emph{augment} human review, not replace it, and humans should then review any concern an AI raises. Such a review is relatively inexpensive and its downside is bounded, while the potential benefit (catching a serious flaw that people missed) can be large.

We have taken our own advice, having two independent AI systems, Google Gemini and Anthropic's Claude Code, review our inputs and the resulting models. This is not a proof, but an additional cross-check. Anyone can verify the models, as we do by hand in Appendix~\ref{sec:appendix}. The AI review adds confidence but the result rests on the checkable models. Gemini reported that the inputs had:

\begin{enumerate}
	\item Proper use of goals: Every model cleanly defined the target independent assumption as a standalone conjecture correctly placed in the goals block and was not in the assumptions block.
	\item No unfinished blocks: There are no missing \code{end_of_list} markers or dangling periods that might enable silent logic corruption.
	\item Correctly enabled Prolog variables: The directive \code{set(prolog_style_variables)} is correctly initialized. This guarantees that any identifier starting with a capital letter (such as \code{Source}) is correctly treated as a variable, while identifiers starting with lowercase letters (such as \code{sP}) are safely locked as constants, per our conventions.
	\item Correct case use: These tools are case-sensitive. Every single constant (like \code{cP}, \code{sP}, \code{lsP}, \code{eArun}) successfully starts with a lowercase letter, and every variable (like \code{Source}, \code{Language}, \code{Input}, \code{Executable1}, \code{Environment1}) starts with an uppercase letter. There are no capitalization mismatches that would break model satisfiability.
\end{enumerate}

For the outputs, Gemini likewise checked that the models Mace4 produced really were valid models. Claude Code independently confirmed both, re-deriving the input checks and re-evaluating each of the 17 models against its formulas, and agreed that every assumption holds while every negated target fails. Using more than one AI reviewer parallels the common practice of using more than one human reviewer.

AI does not replace the need for tools that are designed to guard against common mistakes. An AI reviewer is non-deterministic and offers no guarantees; asked the same question twice, especially in slightly different ways, it may not give the same answer. Also, AI systems are trained largely on human work, so they tend to make the same mistakes that people make. In contrast, a tool that prevents or flags likely errors can do this deterministically, every time. Maximally safe tools, AI review, and human review are complementary layers, each catching what the others can miss. They work best \emph{together}.

\subsection{Proofs and Provers as Tools}

Bertrand Russell stated that mathematics possesses a ``supreme beauty---a beauty cold and austere\ldots''~\cite{russell1910study}. We agree that mathematics can be beautiful.

However, developers of software and systems often need to use mathematics as tools. In particular, when it is \emph{vital} that something be true in the real world, it is vital to be able to quickly get correct and certain answers. Provers that can create proofs, both automatically and interactively, can help achieve that. However, it is vital that provers have the ability to detect and counter likely mistakes, as well as provide simplifying affordances. Otherwise, the ``proof'' they generate may be misunderstood, and people may be put in unexpected danger.

Civil engineers routinely use systems to mathematically analyze models to find dangers. Sometimes software developers also use mathematical tools, such as provers, to verify processes (such as DDC), protocols, and even code. At Amazon Web Services, for example, engineers used the TLA+ specification language and model checking to verify the designs of critical distributed systems, where it found ``subtle bugs we are sure we would not have found by other means'' such as design reviews, code reviews, and testing~\cite{newcombe2015aws}. It is important that these tools be designed to better support such uses.

\section{Conclusion and Next Steps}

By formally proving the independence of the assumptions underlying the three proofs of diverse double-compiling (DDC), we have established that there are no redundant assumptions in the DDC mathematical framework. When practitioners apply DDC to secure their distribution pipelines, they cannot assume that any single condition is a ``freebie'' implied by the others. Each assumption, from compiler determinism to the independence of the environment configurations, must be validated with care. On the other hand, they can be confident that their efforts are not wasted, since each assumption really does need to be verified.

Our experience makes the broader case of this paper: mathematical tools should be designed to be \emph{maximally safe}. The Mace4 scoping trap and missing ending trap underscore that formal tools are only as robust as our understanding of their underlying semantics. Both humans and AI make mistakes. Tools should not silently accept input if it is unlikely to be what was intended. In addition, tools should provide affordances that reduce the likelihood of mistakes. A tool should do as much as it reasonably can to prevent and catch mistakes, not merely the minimum.

This work lays a rigorous foundation, but it is just one step in securing the global digital supply chain. Both threads of this paper serve that goal: diverse double-compiling helps counter attackers, even sophisticated ones, and maximally safe tools let us trust the proofs on which such defenses rest. Ultimately, the goal of this research is that all of us can use software that is increasingly trustworthy.

\subsection{A Shared-Symbol Case: \texttt{cT\_compiles\_sP}}

Unlike the other four assumptions of the key proof, \code{cT_compiles_sP} has no free symbol: every symbol it uses (the relation \code{accurately_translates} and the constants \code{cT}, \code{lsP}, \code{sP}, \code{e1}, and \code{e2}) also appears in another assumption, so its independence cannot be argued from a single unconstrained symbol. Mace4 nonetheless finds a countermodel immediately (Listing~\ref{lst:ct}): the structure in which every constant is $0$, the function \code{compile} is identically $0$, and the relations \code{accurately_translates} and \code{exactly_correspond} are false everywhere.

The negated goal is \code{cT_compiles_sP}, which asserts that the trusted compiler \code{cT} accurately translates the parent source \code{sP} in every environment. To refute a claim about \emph{every} environment it suffices to exhibit one environment in which it fails, so Mace4 names a single candidate environment with a fresh constant \code{c1} (a \emph{Skolem constant}, the standard name for such a witness). Since \code{accurately_translates} is false everywhere in this model, the assumption fails at \code{c1}, so the goal is falsified. The two implication assumptions, \code{define_exactly_correspond} and \code{sP_compiles_sA}, are then satisfied \emph{vacuously}, because their antecedents are \code{accurately_translates} atoms, and that relation never holds here. The two definitions, \code{definition_stage1} and \code{definition_stage2}, each equate a stage constant to a \code{compile} term, and both sides are $0$ in this model: the stage constant because every constant is $0$, and \texttt{compile(0,0,0,0,0)} because \code{compile} is identically $0$. Each definition therefore reduces to the true equation $0 = 0$.

This countermodel is a concrete instance of the vacuous-truth phenomenon discussed in Section~\ref{sec:lessons}: a universally quantified implication is satisfied whenever nothing meets its antecedent. Here that vacuity is exactly what lets the remaining assumptions coexist with the negation of \code{cT_compiles_sP}, confirming its independence.

\subsection{A Shared-Symbol Case with a Substantive Model: \texttt{cP\_corresponds\_to\_sP}}

Our final example is a shared-symbol case whose countermodel, unlike the previous one, is \emph{substantive} rather than vacuous. It negates \code{cP_corresponds_to_sP}, an assumption of the nine-assumption \code{always_equal} proof stating that the parent compiler \code{cP} corresponds to its source \code{sP}. Like \code{cT_compiles_sP}, this assumption has no free symbol. Here, however, Mace4 does not satisfy the remaining assumptions by making everything false. Instead it builds the structure in Listing~\ref{lst:cp}, setting \code{sP} and \code{stage1} to $1$ and populating the \code{compile}, \code{run}, \code{converttext}, \code{retarget}, and \code{extract} tables with non-trivial values.

The goal \texttt{exactly\_correspond(cP, sP, lsP, eA)} evaluates to \texttt{exactly\_correspond(0, 1, 0, 0)}, which is false, so the assumption is falsified. The remaining assumptions are satisfied concretely. Consider \code{definition_stage1}, which requires \code{stage1} to equal \texttt{compile(sP,\allowbreak{} cT,\allowbreak{} e1effects,\allowbreak{} e1,\allowbreak{} e2)}, that is \texttt{compile(1, 0, 0, 0, 0)}. Under \code{define_compile} this value unfolds through the nested tables: \texttt{retarget(1, 0) = 1}, then \texttt{run(0, 1, 0, 0) = 1}, then \texttt{converttext(1, 0, 0) = 1}, then \texttt{extract(1) = 1}. The result, $1$, matches both the \code{compile} table and \code{stage1}, so the assumption holds. Likewise \code{sP_portable_and_deterministic} requires \texttt{portable\_and\_deterministic(sP,\allowbreak{} lsP,\allowbreak{} retarget(sA, eArun))}, which evaluates to \texttt{portable\_and\_deterministic(1, 0, 0)}, an atom that is true in the model.
