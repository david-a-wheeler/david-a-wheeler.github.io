\begin{abstract}
Diverse double-compiling (DDC) counters the classic ``trusting trust'' attack by using a second, trusted compiler to verify the compiler under test. DDC has been formally proved correct, and its assumptions have been proved consistent. However, the DDC assumptions have not been shown to be independent. Independence matters in practice. Applying DDC requires separately verifying that each assumption holds. Proving the assumptions independent shows that none is implied by the others, so none of that verification effort is wasted. This paper demonstrates that the assumptions of all three DDC proofs are independent, constructing a countermodel for each assumption. Along the way, we document subtle pitfalls encountered during automated countermodel generation. These experiences yield broader lessons on the importance of defensive tool engineering and language affordances that mitigate the risk of misinterpreted proof requests.
\end{abstract}

\keywords{diverse double-compiling, DDC, trusting trust, supply chain security, open source software, formal proof, assumption independence}

\section{Introduction}

Most software security issues have, in principle, well-un\-der\-stood solutions:

\begin{enumerate}
	\item Many programs contain unintentional vulnerabilities. These vulnerabilities can in principle be identified using human review, traditional static analysis, traditional dynamic analysis, and artificial intelligence (AI).
	\item Programs may have intentional vulnerabilities in their source code. These can often be detected through similar methods, in particular, by reviewing their source code. Attackers can create ``underhanded'' source code written to try to mislead reviewers, but there are countermeasures against underhanded code that reduce that risk~\cite{wheeler2020underhanded}.
	\item A compiled executable of an ordinary program may not correspond directly to the source code reviewed by others due to causes such as manipulated source releases, subverted builds, subverted distribution, or corruption after loading. Examples include the attack on SolarWinds Orion in 2020~\cite{wheeler2021preventingsolarwinds} and the backdoor in XZ Utils revealed in 2024~\cite{akamai2024xzutils}. Countermeasures exist, including protecting builds (see OpenSSF SLSA~\cite{openssfslsa}), digital signatures (see OpenSSF Sigstore~\cite{openssfsigstore}), and reproducible builds~\cite{reproduciblebuilds}.
\end{enumerate}

Addressing these challenges is not necessarily \emph{easy}, but at least there \emph{are} known ways to address them. Unfortunately, a different and clever attack subverts all of these countermeasures.

\subsection{Trusting Trust Attack}

An Air Force evaluation of Multics by Karger and Schell \cite{karger1974multics,karger2002thirty}, and Ken Thompson's Turing Award lecture (``Reflections on Trusting Trust'') \cite{thompson1984trust}, showed that compilers can be subverted to insert malicious Trojan horses into critical software, including themselves, as part of the compilation process. If this ``trusting trust'' attack goes undetected, even complete analysis of a system's source code will not find the malicious code that is running. Re-running the compiler does no good, because such a compiler can self-subvert. The ability to subvert future versions of tools makes this attack challenging to counter.

For decades, the industry treated the ``trusting trust'' attack as an unresolvable problem. Yet if you cannot completely trust the compiler executable used to build your tools, you cannot trust anything built with it, including other compiler executables descended from its compilation. This is especially important for open source software, where people verify source code but execute compiled executables.

\subsection{Countering the Trusting Trust Attack}

Diverse double-compiling (DDC) counters the trusting trust attack by using a second, trusted compiler to compile the parent of the compiler under test, then using that result to compile the source code of the compiler under test \cite{wheeler2010ddc}. If the output of this process matches the original compiler executable, then the executable corresponds to its source code. DDC does not require that arbitrary different compilers produce the same executable output given the same input. Indeed, this would be extremely unlikely in the real world. DDC instead requires determinism by the parent compiler of the compiler under test. It is generally desirable that a compiler be deterministic, and many real-world compilers are deterministic or can be made deterministic via a flag.

Another countermeasure against the trusting trust attack is to implement a bootstrappable build~\cite{bootstrappableorg}. Bootstrappable builds work to minimize the number and size of bootstrap binaries, which are then used to build the next layer, which in turn builds the next, and so on until everything else is built. This is a promising approach, but challenging, and many environments are not based on bootstrappable builds. DDC and bootstrappable builds are strongly complementary approaches. Bootstrappable builds limit opaque binaries and strongly address the question of how to get a trustworthy trusted compiler. DDC can use those resulting bootstrapped builds to verify other results that were \emph{not} originally created with bootstrappable builds.

DDC has been formally (mathematically) proved to produce this result when a set of reasonable assumptions is true \cite{wheeler2010ddc}. The assumptions and goals were defined in first-order logic (FOL), and proofs were generated with Prover9 \cite{mccune2009prover9}. There is strong confidence that these proofs are correct; they were automatically generated, checked step-by-step by a different formally-proven tool, and reviewed by multiple people. The same source also proved that these assumptions were consistent (that is, they did not contradict each other).

The proofs use a systematic notation for the compilers, source code, and intermediate results involved, following the DDC formalization~\cite{wheeler2010ddc}. A symbol's leading letter gives the kind of object (\code{c}, compiler executable; \code{s}, source code; \code{l}, source language; \code{e}, environment) and its suffix gives the role (\code{A}, the compiler under test; \code{P}, its parent; \code{T}, trusted). In actuality, there are three proofs involving DDC:

\begin{enumerate}
	\item Goal \code{source_corresponds_to_executable} (5 assumptions): This is the key proof for DDC. It shows that given certain assumptions, if \code{stage2} (the result of the DDC process) and \code{cA} (the original compiler-under-test) are equal, then the executable \code{cA} and the source code \code{sA} exactly correspond. (Filename prefix \code{ddc}.)
	\item Goal \code{always_equal} (9 assumptions): This proves that, under ``normal conditions'' (such as when compiler executables have not been rigged and thus do correspond to their respective source code), \code{cA} and \code{stage2} are in fact always equal. Thus, the first proof is actually useful, because its assumptions will often hold. This also implies that if \code{cA} and \code{stage2} are not equal, then at least one of its assumptions is not true. (Filename prefix \code{all-repeats}.)
	\item Goal \code{cP_corresponds_to_sP} (3 assumptions): The previous ``\code{always_equal}'' proof does not require that a ``grandparent'' compiler exist, but having one is a common circumstance. This third proof shows that if there is a grandparent compiler, one of the assumptions of proof \#2 can be proved given other assumptions that may be easier to verify (potentially making DDC even easier to apply in this common case). (Filename prefix \code{cP-corresponds-sP}.)
\end{enumerate}

This paper makes two contributions. First, we prove that the assumptions of all three DDC proofs are independent, so none of them is redundant and none of the effort spent verifying them is wasted. Second, in establishing this result we encountered subtle ways in which automated reasoning tools can silently misinterpret a proof request, and we distill these into lessons for designing tools that are harder to misuse. The two are connected: we could obtain the independence result with confidence only because we recognized and avoided these pitfalls, so the formal result and the tool-safety lessons are two outcomes of the same careful process.

\section{Related Work}

The trusting trust attack remains a recognized threat to software supply chain integrity. Reproducible builds confirm that a binary corresponds to its source, but they assume a trustworthy compiler and so do not by themselves counter a subverted compiler~\cite{lamb2022reproducible}. Cox has since reconstructed and annotated Thompson's original self-re\-pro\-duc\-ing compiler, making the attack concrete for a modern audience~\cite{cox2023nih}.

Diverse double-compiling was briefly suggested by Henry Spencer and then developed, named, and demonstrated by Wheeler, who introduced DDC~\cite{wheeler2005ddc}. Wheeler later generalized DDC to support arbitrary parent compilers and gave machine-checked proofs of its correctness and of the consistency of its assumptions~\cite{wheeler2010ddc}. Skrimstad implemented a self-rep\-li\-cat\-ing attack against the Go compiler and proposed applying DDC with more than one trusted compiler to raise assurance beyond applying DDC only once~\cite{skrimstad2018ddc}. Rosencrantz applied DDC to harden cryptocurrency software against supply-chain attacks~\cite{rosencrantz2023ddc}. Christensen integrated DDC into a CI/CD pipeline so that compiler verification runs automatically as part of the build workflow~\cite{christensen2025ddc}. More broadly, the re\-pro\-duc\-ible-builds community has rebuilt bootstrap compilers such as GNU Mes bit-for-bit across independent distributions and toolchains, which could be used as a trusted compiler for applying DDC~\cite{mesbootstrap2019}. None of these papers proved that the DDC assumptions are \emph{independent}.

\section{Assumption Independence in First-Order Logic}

If a proof uses a set of assumptions, that does \emph{not} mean that the assumptions are independent. It is possible that some of its assumptions can be derived from some of the other assumptions. This is not desirable. When applying DDC in practice it takes effort to gain strong confidence that each assumption is true. It would be better if we could either (1) prove that some assumptions are derived from others (in which case they no longer need to be separate assumptions), or (2) prove that all the assumptions are independent of each other in their respective proofs.

Given some set of consistent assumptions $Q$, and another assumption $P$, then $P$ is independent of $Q$ if both $Q \cup \{P\}$ and $Q \cup \{\neg P\}$ yield consistent assumptions. This is noted, for example, by Kunen~\cite[page xi]{kunen1980set}. It is already formally proved that our entire base set of DDC assumptions is consistent, so the first half of that definition ($Q \cup \{P\}$) is already proved. Therefore, to prove $P$'s independence, we only have to show that $Q \cup \{\neg P\}$ is consistent.

In first-order logic, the easiest way to show that a set of assumptions is consistent is to find a ``model'' where they are all true. Formulas in first-order logic contain abstract syntax symbols (variables, relations, and functions) with no inherent mathematical meaning. A \emph{structure} (or \emph{interpretation}) defines a specific (``concrete'') domain of objects (aka a universe of discourse) and maps these abstract symbols to actual relations, functions, and constants on that domain. An \emph{assignment} (or \emph{valuation}) then maps the free variables to specific elements within that domain. Under this interpretation and assignment, we evaluate the truth value of the formulas. A structure is a \emph{model} of a set of formulas if we can assign concrete values from the domain to the variables such that every formula in the set evaluates to true~\cite{enderton2001logic}.

A model is not a claim about absolute, universal truth across all possible universes. Rather, it is an assertion of \emph{satisfiability}. It is a concrete, mathematically consistent instance (a specific ``world'' with specific assignments of values) where the assumptions are satisfied. To prove independence, we do not need to show that our assumptions are universally true; we just need to show that there exists at least one such satisfying assignment of values (a model) where $P$ is false while all other assumptions in $Q$ remain true. In short: for each assumption $P$, we just need to find a model where $P$ is false but all other assumptions in $Q$ are true.

Hesseling explains that Heyting~\cite{heyting1930regeln} uses this exact approach to prove the independence of his formalization of intuitionistic logic, and credits Bernays with the technique in which ``what we would nowadays call models in which each time ten of the eleven given axioms are true, the eleventh one false\dots\ [by] varying the eleventh axiom and presenting a different model to which the new combination of axioms applies, he proves that all the axioms are independent of each other''~\cite[page 278]{hesseling2003gnomes}.

\section{Our Approach for Finding Models}

To formally establish the independence of the DDC assumptions, we used automated logic tools. Specifically, to show that a set of $n$ assumptions is independent for some proof, we must prove that no single assumption can be derived from the remaining $n-1$ assumptions. We can do that by demonstrating a counterexample.

In our case, for each assumption $a_i$, we must show that there exists a mathematically consistent world where $a_i$ is false, yet all other assumptions in a given proof remain true. If we do this for each assumption for all three proofs, we have shown that all of the assumptions are independent of each other. More precisely, each assumption is independent of all the others taken together (their conjunction), which is strictly stronger than pairwise independence.

We worked to accomplish this by feeding our formal assumptions into Mace4~\cite{mccune2009prover9}, which searches for finite mathematical models. Mace4 is not at the cutting edge of model creation technology today. However, it is still a fine tool, and Mace4 takes the same syntax as the Prover9 tool which was used to generate the original proofs.\footnote{A maintained modernization, LADR-2026 by Machado and Lesyna, has since been released. It builds on the original code, and preserves McCune's algorithms and input syntax, while repairing and extending the tools (for example, native TPTP/TSTP support)~\cite{machado2026ladr}. We deliberately used the same version as in the original DDC proofs, to avoid any behavioral differences. However, this actively maintained version is a natural choice for new work.} Using the tool specifically designed to take the syntax avoided the risk of mistranslating the statements if we switched to a different system.

\subsection{The Mace4 Variable Scope Trap}

When attempting to formally prove independence using Mace4, early drafts ran into a subtle roadblock while analyzing the second assumption of goal \code{always_equal}. This occurred with the assumption defining a portable and deterministic program, labeled \code{define_portable_and_deterministic}.

Consider this attempted negation of the assumption, where we take the original assertion and wrap it with \texttt{- ( ... )}:

\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={An attempted negation that wraps the assumption in \texttt{- ( ... )}, which Mace4 misinterprets.}]
formulas(assumptions).
  - ( ( portable_and_deterministic(Source,
          Language, Input) &
        exactly_correspond(Executable1, Source,
          Language, Environment1) &
        exactly_correspond(Executable2, Source,
          Language, Environment2)) ->
          ( converttext(run(Executable1, Input,
              EnvEffects1, Environment1),
              Environment1, Target) =
            converttext(run(Executable2, Input,
              EnvEffects2, Environment2),
              Environment2, Target)))
    # label(define_portable_and_deterministic).
end_of_list.
\end{lstlisting}
\end{minipage}

At first glance, this looks completely reasonable. We want the negation of the rule, so we put a negation symbol ($-$) in front of it, and we surround everything else with parentheses to ensure it has the highest precedence. Unfortunately, this creates a catastrophic misinterpretation due to how Mace4 handles unquantified variables.

In Prover9 and Mace4 set to Prolog-style variables, when you use capitalized terms (like \code{Source}, \code{Language}, or \code{Input}) without an explicit quantifier, the tool treats them as free variables. To process the file, the engine must bind these variables. It does this by automatically wrapping the entire expression in an implicit universal quantifier ($\forall$). This made sense for the original assumption. However, because in this modified assertion the negation symbol ($-$) is written \emph{inside} the statement, Mace4 applied the implicit universal quantifier \emph{outside} the negation. Mathematically, Mace4 interpreted the assumption as:
\[\forall \vec{x} \, \neg (\text{Premise}(\vec{x}) \rightarrow \text{Conclusion}(\vec{x})).\]

For an implication ($\rightarrow$) to be false, the premise must be true \emph{and} the conclusion must be false. Because of the mis-applied universal quantifier, the resulting request was to satisfy an absurd set of requirements for all of the objects covered in the implicit universal quantifier.

To prove independence, we simply want to prove that there is \emph{at least one} counterexample. Mathematically, a true counterexample requires an existential quantifier ($\exists$) wrapped around the negation: $\exists \vec{x} \, (\text{Premise}(\vec{x}) \land \neg \text{Conclusion}(\vec{x}))$.

In raw Mace4 syntax you can manually write out every single variable to force this behavior:

\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={Tediously forcing the intended existential by naming every variable by hand.}]
exists Source Language Input Executable1
       Environment1 Executable2 Environment2
  EnvEffects1 EnvEffects2 Target (
    portable_and_deterministic(Source, Language,
      Input) &
    exactly_correspond(Executable1, Source,
      Language, Environment1) &
    exactly_correspond(Executable2, Source,
      Language, Environment2) &
    - ( converttext(run(Executable1, Input,
          EnvEffects1, Environment1),
          Environment1, Target) =
        converttext(run(Executable2, Input,
          EnvEffects2, Environment2),
          Environment2, Target))
).
\end{lstlisting}
\end{minipage}

Writing this out manually is tedious and highly susceptible to typos or missed variables, which raised some concerns.

We then realized there was a much safer en\-gi\-neer\-ing-focused solution: let the tool do the negation for us, including handling the quantifiers. Instead of manually negating the rule in the assumptions list, we simply moved the statement into a new \code{formulas(goals)} block:

\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={The safe alternative: move the target assumption into a \texttt{formulas(goals)} block.}]
formulas(assumptions).
    % All other DDC assumptions unmodified
end_of_list.

formulas(goals).
    % The assumption to be shown independent
end_of_list.
\end{lstlisting}
\end{minipage}

When Prover9 or Mace4 sees a statement in the \code{goals} block, it uses a built-in automated mechanism that correctly negates the expression behind the scenes. By using the \code{goals} block, we eliminate the risk of quantifier mismatch. Mace4 will search for a model that satisfies all our assumptions while breaking the goal in exactly \emph{one} valid place. If it finds that model, our independence proof is correct, our syntax remains clean, and we avoid subtle scoping traps.

The \code{goals} block is an example of a \emph{syntactic affordance}, that is, a syntactic construct designed to make common constructs easier to \emph{correctly} express. The term \code{goals} is a little misleading, and stems from the fact that Prover9 is a ref\-u\-ta\-tion-based theorem prover. It would be even nicer if \code{goals} had a synonym like \code{negated} when you are not really expressing a goal. Still, the mechanism is a significant aid.

However, using this approach to avoid a trap risks triggering \emph{another} trap.

\subsection{The Missing Ending Trap}

Typically, goals follow assumptions in these files, using the
\code{formulas(goals)}
header block to open the section. But what happens in Mace4 if the assumptions list is not terminated with an explicit \code{end_of_list} separator?

The answer, unfortunately, is that this is a major hazard. This tiny syntactic omission can silently change the entire semantic meaning of the proof request without throwing a parser error or even a warning. This behavior creates a dangerous ``gulf of evaluation'' where a human reader interprets the input file one way, but the tools interpret it in a completely different way.

Consider a typical Prover9 or Mace4 input file where a user defines a set of background assumptions and a target goal to test for independence:

% Use minipage to force this listing to stay in 1 column (skipping to
% the next column if it must). It's short, and splitting the listing
% across columns makes it ugly.
\noindent
\begin{minipage}{\columnwidth}
\begin{lstlisting}[language=Prolog, caption={A problematic LADR input block containing a missing boundary separator.}]
formulas(assumptions).
  assumption_1.
  assumption_2.
  % Missing 'end_of_list.'

formulas(goals).
  target_goal.
end_of_list.
\end{lstlisting}
\end{minipage}

To a human eye, the block boundary is obvious. We see \code{formulas(goals)} and naturally assume the tool transitions from parsing assumptions to parsing goals. Many systems do work this way, for example, an HTML processor that reads a list item start (\code{<li>}) will close the previous list item. In the LADR parser used by Mace4 and Prover9, however, \textbf{no such transition occurs}.

The silent semantic shifting of formulas occurs due to specific implementation properties of the sequential engine:

\begin{enumerate}
    \item \textbf{Sequential Term Parsing:} The LADR parser is a sequential engine. When it enters a \code{formulas(list_name).} block, it continuously parses incoming lines as first-order terms (formulas) until it hits the specific, isolated atomic symbol \code{end_of_list}.
    \item \textbf{Functor Misinterpretation:} Because \code{"formulas"} is not a reserved keyword in the lexer, the line \code{formulas(goals)} is parsed as a syntactically valid \emph{compound term} (specifically, a functor of arity 1 with the constant argument \code{"goals"}).
    \item \textbf{Silent Appending:} Because there was no \code{end_of_list} to break the parsing loop, the parser simply appends the compound formula \code{formulas(goals)} to the active \code{assumptions} list.
    \item \textbf{Goal Hijacking:} Every subsequent line, \emph{including} the intended \code{target_goal}, continues to be parsed as an assumption. The parser only exits the assumption list when it finally reaches the single \code{end_of_list} at the very bottom of the file.
\end{enumerate}

This behavior leads to a silent, compounding failure that misleads both the tool and the investigator. In our case, the tool immediately found a model that solved the misinterpreted problem. The investigator, seeing a model generated, falsely concludes \emph{``Excellent, Mace4 found a valid countermodel, so my assumption is independent!''} In reality, the tool proved nothing useful when \code{end_of_list} was missing.

\section{Results}

Once these problems were addressed, Mace4 successfully identified models for all 17 ($5+9+3$) situations. Each model is a countermodel showing a way to satisfy the negation of the target assumption while also satisfying all other background assumptions within its respective proof.

This formally proves that \emph{none} of the DDC assumptions are redundant within their proofs. All 17 models found have a domain size of 2. This small size is not a theoretical constraint, but it does suggest it should be relatively easy for many other automated model finders to find at least one relevant model.

\subsection{Discussion}

Each model, except one, was found in less than a second. The exception was \code{all-repeats-independence-7}, which required 42 seconds of search time. Search time is not important compared to the mathematical confirmation of satisfiability, but this demonstrates the computational feasibility of the check.

There is a good reason that finding the model for \code{all-repeats-independence-7} took longer. This model required negating the assumption \code{definition_cA} which states \texttt{cA = compile(sA, cP, eAeffects, eA, eArun)}. Mace4 searches for a model by assigning values to the cells of each function and relation table over the finite domain. This assumption defines the \code{compile} function as a deeply nested mapping:
\texttt{compile(S, C,\allowbreak{} E,\allowbreak{} R, T) =\allowbreak{} extract(\allowbreak{}converttext(\allowbreak{}run(C,\allowbreak{} retarget(S, T),\allowbreak{} E, R),\allowbreak{} R, T))}.
This definition involves a 5-ary function (\code{compile}), a 4-ary function (\code{run}), and a 3-ary function (\code{converttext}). With our small domain size of $D = 2$, the 3-ary function is not too bad; it only has $2^{(2^3)} = 2^{8} = 256$ possible configurations. However, even with our small domain size of $D = 2$, a 5-ary function has $2^{(2^5)} = 2^{32} = 4{,}294{,}967{,}296$ possible configurations, and a 4-ary function has $2^{16} = 65{,}536$ combinations.

When Mace4 negates the goal (\code{cA} $\neq$ \code{compile(...)}), it must satisfy all the other relevant assumptions. Because the other assumptions contain specific instances of \code{compile} (such as \code{stage1} and \code{stage2}), Mace4 must construct a consistent mathematical universe across this massive combinatoric space of 5-ary and 4-ary constraints (as well as the 3-ary constraint). The 42-second search time reflects the size of this constraint space (the shared \code{compile} table that every assumption in this proof constrains), not any error or special depth in this particular independence result.

Independence here takes two forms, distinguished by a simple syntactic test: whether the negated assumption rests on a \emph{free} symbol or only on \emph{shared} ones. In 11 of the 17 cases the assumption contains a symbol (a constant, function, or relation) that occurs in no other assumption of its proof; examples include the constants \code{stage2}, \code{cA}, and \code{e1effects}, and the \code{extract} function. A countermodel can then falsify that assumption by interpreting its free symbol however it likes, so the independence is immediate. In the remaining 6 cases every symbol is shared with another assumption, so no such one-symbol argument applies. This syntactic split does not track search difficulty. The single model that required a long search was the 42-second \code{definition_cA} case above, and it negates one of the 11 free-symbol assumptions; its cost comes from the shared background theory, not from any depth in that assumption's independence. Either way, no assumption is derivable from the others. Because independence concerns derivability rather than ease of checking, even an assumption that is independent for such a trivial reason must still be separately verified to best assure ourselves that the DDC assumptions are independent.

We could find models using \emph{other} tools, such as Z3 or cvc5, but that is not necessary. If \emph{any} tool finds a model, then a model has been found. We primarily chose Mace4 to avoid translation concerns, but there were other good reasons to use Mace4 for this specific task. Modern SMT solvers like Z3 and cvc5 are optimized to prove unsatisfiability, so they can struggle when searching for satisfying models with first-order quantifiers~\cite{demoura2011smt,reynolds2013fmf}. Mace4 bypasses this quan\-ti\-fi\-er-reasoning difficulty entirely by systematically searching small, finite domains starting at size 2, turning a computationally unbounded search into a simple, highly solvable constraint problem. That said, we expect other tools \emph{could} find models, and that might be interesting future work.

Since we have proved that the assumptions are independent, application of DDC that relies on the published proofs cannot take a shortcut. To rely on the security guarantees of any of the DDC proofs in \cite{wheeler2010ddc}, you must establish the validity of all of the assumptions of the proofs you depend on.

\subsection{Limitations}

Our results have a few natural limitations. First, independence is established relative to the specific first-order formalization of DDC given in~\cite{wheeler2010ddc}. A different but equivalent formalization of the same process could combine or split assumptions, and might therefore exhibit different dependencies; our claims concern the DDC assumptions exactly as they are published. Second, establishing independence requires only a single countermodel per assumption, so we do not claim that the particular models we found are canonical, minimal, or unique; they serve only as witnesses that the required countermodels exist. Third, independence is a statement about derivability alone. It confirms that each assumption must be verified separately, but it says nothing about how difficult any given assumption is to establish for a real compiler; an assumption that is trivially independent may still be the hardest to verify in practice.

\section{Lessons Learned}
\label{sec:lessons}

These experiences highlighted some lessons that should be learned.

\subsection{Mathematical Tools Should Be Maximally Safe}

Modern cars do not just go somewhere. They are designed to alert users about unusual events, to attempt to protect people if a mistake is made, and to make it easy to perform typical tasks in a safe way. Modern cars include engine lights, turn and brake signals, seat belts, air bags, bumpers, speedometers, and \emph{many} other features to alert, protect, and simplify correct use. We cannot prevent all collisions, but we can reduce their likelihood and expected impact. Cars whose designs ignore these requirements are unsafe at any speed~\cite{nader1965unsafe}.

Similarly, mathematical tools should be designed to alert on unusual events, attempt to protect their users from mistakes, and simplify correct use. Both humans and AI make mistakes, often following patterns that are easy to predict. There are also common uses that, if made easy, are more likely to be done correctly. A tool need not have \emph{many} such mechanisms, but in practice a good one often will. Tool makers may omit them from an ill-conceived and narrow conception of the tool's requirements, dismissing such safeguards as irrelevant extras, \emph{exactly} as early car makers once dismissed seat belts and the like. We cannot prevent all mistakes with these tools, but \emph{can} reduce their likelihood and expected impact, while also making them easier to safely use.

An old proverb says, ``A bad workman always blames his tools''. The proverb is true but often misunderstood; a wise worker knows to avoid or fix bad tools.

\subsubsection{Examples from This Experience}

There is more than one way to address the variable scope trap. The system could require \emph{all} variables to be bound, for example. Another solution would be to alert on suspicious patterns like a leading \code{-} of a formula with an unbound variable. Once the problem is detected, Mace4 does have the affordance \code{formulas(goals)} to reduce the risk of incorrect negation. That is a welcome capability, though alternative clear names would be an improvement.

Similarly, there is more than one way to deal with a missing ending. The tool could reject a statement starting a new section when a section is already open (this is the approach we prefer). The tool could also quietly close any section in progress when it is asked to start a new section. The point is that the \emph{current} behavior is dangerous.

\subsubsection{Similar Tools}

This lesson that mathematical tools should be designed to be maximally safe is not specific to Mace4. It also applies to other similar tools such as Z3 and cvc5, both of which support the SMT-LIB format. For example, Mace4 and tools supporting SMT-LIB format handle division-by-zero in a non-intuitive and dangerous way. In SMT-LIB, division is defined as a total function, which means $x / 0$ has a meaning yet it has no constraints on its value~\cite{BarFT-RR-25}. Mace4 behaves similarly: rather than rejecting a division (or \code{mod}) by zero, it treats such an expression as one that can take whatever value leads to a solution, so $x / 0$ is effectively unconstrained~\cite{mccune2009mace4arithmetic}. Failing to explicitly constrain the divisor to be non-zero can produce misleading ``models'' that appear to provide a ``solution''. Yet tools supporting SMT-LIB generally do not automatically add the ``do not divide by 0'' constraint, nor do they warn about it.

\subsubsection{Exists with Implies}

We can even find risky constructs in the underlying first-order logic notation that many systems are built on. In some ways, these are the most dangerous and need the most emphasis. \emph{All} tools that build on this widely-used notation, and do not take steps to deal with them, can lead to the same problems.

A common translation error involves the existential quantifier (``exists'') paired with material implication (``implies''), structurally written as:
\[ \exists x (P(x) \rightarrow Q(x)). \]
This is almost always a catastrophic misstatement of user intent. Typically, an investigator intends to assert that there exists some object $x$ which possesses property $P$ and satisfies consequence $Q$, which must be formally stated using a conjunction:
\[ \exists x (P(x) \land Q(x)). \]
The implication form is dangerous because $P(x) \rightarrow Q(x)$ is equivalent to $\neg P(x) \lor Q(x)$, so any object that is not a $P$ satisfies it vacuously. A solver can thus ``confirm'' that a malicious program exists by pointing at something that is not a program at all. The existential quantifier pairs naturally with conjunction, whereas the universal quantifier pairs with implication~\cite{barwise2011lpl}.

The SPARK system has an option that warns about this construct (among others) because it is such a common mistake~\cite{wheeler2026allsome}.

\subsubsection{Combining Exists and For-All: Allsome}

In standard first-order logic, universal statements of the form $\forall x (P(x) \rightarrow Q(x))$ are susceptible to the trap of \emph{vacuous truth}. If no object in the universe satisfies the antecedent $P(x)$, the entire universal implication always evaluates to true. Put plainly, both ``all martians are green'' and ``all martians are not green'' are true if there are no martians. In security verification, this can be disastrous.

To prevent this error, an investigator must manually chain an existential prerequisite to the universal claim:
\[(\exists x P(x)) \land (\forall x (P(x) \rightarrow Q(x))).\]
This is mathematically sound, but writing this compound expression repeatedly in raw tool syntax introduces a heavy unnecessary cognitive load. It also invites typos, variable mismatch, or scoping errors.

The ``allsome'' quantifier is a helpful affordance in such cases. A language that supports this quantifier simplifies guaranteeing that at least one instance of the target subject exists (\emph{some}) and that every instance satisfies the given criteria (\emph{all}). This defensive language feature eliminates the risk of vacuous satisfaction while keeping the proof source clean, readable, and less error-prone~\cite{wheeler2026allsome}. The ``allsome'' quantifier has been defined, and its key properties proved, in the Metamath Proof Explorer (the \code{set.mm} database)~\cite{metamathmpe}, but it would be good to see it implemented in more tools.

\subsection{Use AI to Review}

Both of the problems listed above were detected by AI, specifically Google Gemini. Mace4 and Prover9, along with their documentation, are widely available and have been for years, so many well-trained AI systems already knew about these tools and their foibles.

We think it is important today to have at least one AI review important proofs involving the physical world, in addition to human review.


While this is only one set of proofs, it demonstrates that AI \emph{can} sometimes find important problems that human reviewers might miss. This must be kept in context: AI also generates false claims, expresses confidence in false statements, and offers no guarantee of finding every problem. Depending \emph{only} on AI review would be a mistake. AI review should \emph{augment} human review, not replace it, and humans should then review any concern an AI raises.

We have taken our own advice. All of these inputs were also reviewed by Google Gemini. It determined that these inputs had:

\begin{enumerate}
	\item Proper use of goals: Every model cleanly defined the target independent assumption as a standalone conjecture in the goals block. This forces Mace4 to search for a genuine countermodel where that assumption is false but all others are true.
	\item Goal isolation: The target assumption was completely omitted from the assumptions block and cleanly declared inside \code{formulas(goals)} at the end.
	\item No unfinished blocks: There are no missing \code{end_of_list} markers or dangling periods that might enable silent logic corruption.
	\item Correctly enabled Prolog variables: The directive \code{set(prolog_style_variables)} is correctly initialized. This guarantees that any identifier starting with a capital letter (such as \code{Source}, \code{Language}, \code{Input}, and \code{Executable1}) is treated as a variable, while identifiers starting with lowercase letters (such as \code{sP}, \code{lsP}, \code{cT}) are safely locked as constants. This was the convention we chose.
	\item Correct case use: These tools are case-sensitive. Every single constant (like \code{cP}, \code{sP}, \code{lsP}, \code{eArun}) successfully starts with a lowercase letter, and every variable (like \code{Source}, \code{Language}, \code{Input}, \code{Executable1}, \code{Environment1}) starts with an uppercase letter. There are no capitalization mismatches that would break model satisfiability.
\end{enumerate}

We also had the AI review all the outputs, to double-check that the models produced by Mace4 really were valid models. This is not a proof, but an additional cross-check. Each model is a concrete finite structure whose validity anyone can verify mechanically, as we do by hand in Appendix~\ref{sec:appendix}; the AI review adds confidence, but the result rests on the checkable models, not on the review.

\subsection{Proofs and Provers as Tools}

Bertrand Russell stated that mathematics possesses a ``supreme beauty---a beauty cold and austere\ldots''~\cite{russell1910study}. We agree that mathematics can be beautiful.

However, developers of software and systems often need to use mathematics as tools. In particular, when it is \emph{vital} that something be true in the real world, it is vital to be able to quickly get correct and certain answers. Provers that can create proofs, both automatically and interactively, can help achieve that. However, it is vital that provers have the ability to detect and counter likely mistakes, as well as provide simplifying affordances. Otherwise, the ``proof'' they generate may be misunderstood, and people may be put in unexpected danger.

Civil engineers routinely use systems to mathematically analyze models to find dangers. Sometimes software developers also use mathematical tools, such as provers, to verify processes (such as DDC), protocols, and even code. It is important that these tools be designed to better support such uses.

\section{Conclusion and Next Steps}

By formally proving the independence of the assumptions underlying the three proofs of diverse double-compiling (DDC), we have established that there are no redundant assumptions in the DDC mathematical framework. When practitioners apply DDC to secure their binary distribution pipelines, they cannot assume that any single condition is a ``freebie'' implied by the others. Each assumption, from compiler determinism to the independence of the environment configurations, must be validated with care. On the other hand, they can be confident that their efforts are not wasted, since each assumption really does need to be verified.

The Mace4 scoping trap and missing ending trap underscores that formal tools are only as robust as our understanding of their underlying semantics. Both humans and AI make mistakes. Tools should not silently accept input if it is unlikely to be what was intended. Tools should avoid being too permissive. In addition, tools should provide affordances that reduce the likelihood of mistakes.

This work lays a rigorous foundation, but it is just one step in securing the global digital supply chain. Ultimately, the goal of this research is to help counter attackers, even sophisticated attackers, so that all of us can use software that is increasingly trustworthy.

\subsection{A Free-Symbol Case: \texttt{definition\_stage1}}

We present a shortened model generated by Mace4 for the file \code{ddc-independence-4.in}. This run establishes the independence of the fourth assumption, the intermediate compiler definition (\code{definition_stage1}), within the key DDC proof. Its free symbol is \code{e1effects}, the parent compiler's build-time environment effects, which occurs in no other assumption of this proof.

This model targets \code{stage1}, the assumption defining the intermediate compiler build step that compiles the parent compiler. This model presents a world where the background definition of \code{stage2} is successfully satisfied, meaning \texttt{stage2 = \allowbreak{}compile(sA, \allowbreak{}stage1, \allowbreak{}e2effects, \allowbreak{}e2, \allowbreak{}eArun)} implies $1 = 1$ (\emph{true}). Yet, in this same world, the definition of \code{stage1} is violated, meaning \texttt{stage1 = \allowbreak{}compile(sP, \allowbreak{}cT, \allowbreak{}e1effects, \allowbreak{}e1, \allowbreak{}e2)} implies $0 = 1$ (\emph{false}). While a practitioner might occasionally obtain a secure compiler through luck or an alternative pipeline, this model demonstrates that without this specific assumption, the security guarantee of DDC no longer logically follows. Without this assumption, we lose the mathematical certainty that the final executable corresponds to its source code, proving that verifying this intermediate step is strictly necessary.

Under this model of domain size 2 (where the domain elements are $\{0, 1\}$), all other background DDC assumptions evaluate to \emph{true}, while the target goal (\code{definition_stage1}) evaluates to \emph{false}.

\subsubsection{Verification in the Model}
Under this model, we can manually check that the background DDC assumptions are satisfied while the target definition is falsified:
\begin{enumerate}
    \item \textbf{Falsification of the Goal:} The target assumption requires that \texttt{stage1 = compile(sP,\allowbreak{} cT,\allowbreak{} e1effects,\allowbreak{} e1,\allowbreak{} e2)}. Substituting the constant valuations from the model yields \texttt{0 = compile(0, 0,\allowbreak{} 0,\allowbreak{} 0, 0)}.
    However, the lookup table indicates that \texttt{compile(0, 0,\allowbreak{} 0,\allowbreak{} 0, 0) = 1}. Since $0 \neq 1$, the goal evaluates to \textbf{\emph{false}}.
    \item \textbf{Satisfaction of the Stage 2 Assumption:} The background assumption requires that \texttt{stage2 =\allowbreak{} compile(sA,\allowbreak{} stage1,\allowbreak{} e2effects, e2,\allowbreak{} eArun)}. Substituting the valuations yields \texttt{1 = compile(0,\allowbreak{} 0,\allowbreak{} 0,\allowbreak{} 0,\allowbreak{} 0)}.
    Looking up \texttt{compile(0,\allowbreak{} 0,\allowbreak{} 0,\allowbreak{} 0,\allowbreak{} 0)} evaluates to $1$. Since $1 = 1$, this assumption evaluates to \textbf{\emph{true}}.
\end{enumerate}

Since all other background assumptions similarly evaluate to true, this structure serves as a mathematically sound counterexample proving that the definition of \code{stage1} is logically independent and cannot be derived from the other assumptions.

\subsection{A Shared-Symbol Case: \texttt{cT\_compiles\_sP}}

Unlike the other four assumptions of the key proof, \code{cT_compiles_sP} has no free symbol: every symbol it uses (the relation \code{accurately_translates} and the constants \code{cT}, \code{lsP}, \code{sP}, \code{e1}, and \code{e2}) also appears in another assumption, so its independence cannot be argued from a single unconstrained symbol. Mace4 nonetheless finds a countermodel immediately (Listing~\ref{lst:ct}): the structure in which every constant is $0$, the function \code{compile} is identically $0$, and the relations \code{accurately_translates} and \code{exactly_correspond} are false everywhere.

The negated goal is \code{cT_compiles_sP}, which asserts that the trusted compiler \code{cT} accurately translates the parent source \code{sP} in every environment. Mace4 introduces a Skolem constant \code{c1} to witness the quantified environment. Since \code{accurately_translates} is false everywhere in this model, the assumption fails at that witness, so the goal is falsified. The two implication assumptions, \code{define_exactly_correspond} and \code{sP_compiles_sA}, are then satisfied \emph{vacuously}, because their antecedents are \code{accurately_translates} atoms, and that relation never holds here. The two definitions, \code{definition_stage1} and \code{definition_stage2}, hold because each reduces to \texttt{0 =\allowbreak{} compile(0,0,0,0,0) = 0}.

This countermodel is a concrete instance of the vacuous-truth phenomenon discussed in Section~\ref{sec:lessons}: a universally quantified implication is satisfied whenever nothing meets its antecedent. Here that vacuity is exactly what lets the remaining assumptions coexist with the negation of \code{cT_compiles_sP}, confirming its independence.

\subsection{A Shared-Symbol Case with a Substantive Model: \texttt{cP\_corresponds\_to\_sP}}

Our final example is a shared-symbol case whose countermodel, unlike the previous one, is \emph{substantive} rather than vacuous. It negates \code{cP_corresponds_to_sP}, an assumption of the nine-assumption \code{always_equal} proof stating that the parent compiler \code{cP} corresponds to its source \code{sP}. Like \code{cT_compiles_sP}, this assumption has no free symbol. Here, however, Mace4 does not satisfy the remaining assumptions by making everything false. Instead it builds the structure in Listing~\ref{lst:cp}, setting \code{sP} and \code{stage1} to $1$ and populating the \code{compile}, \code{run}, \code{converttext}, \code{retarget}, and \code{extract} tables with non-trivial values.

The goal \texttt{exactly\_correspond(cP, sP, lsP, eA)} evaluates to \texttt{exactly\_correspond(0, 1, 0, 0)}, which is false, so the assumption is falsified. The remaining assumptions are satisfied concretely. Consider \code{definition_stage1}, which requires \code{stage1} to equal \texttt{compile(sP,\allowbreak{} cT,\allowbreak{} e1effects,\allowbreak{} e1,\allowbreak{} e2)}, that is \texttt{compile(1, 0, 0, 0, 0)}. Under \code{define_compile} this value unfolds through the nested tables: \texttt{retarget(1, 0) = 1}, then \texttt{run(0, 1, 0, 0) = 1}, then \texttt{converttext(1, 0, 0) = 1}, then \texttt{extract(1) = 1}. The result, $1$, matches both the \code{compile} table and \code{stage1}, so the assumption holds. Likewise \code{sP_portable_and_deterministic} requires \texttt{portable\_and\_deterministic(sP,\allowbreak{} lsP,\allowbreak{} retarget(sA, eArun))}, which evaluates to \texttt{portable\_and\_deterministic(1, 0, 0)}, an atom that is true in the model.

Because Mace4 must find \code{compile}, \code{run}, \code{converttext}, \code{retarget}, and \code{extract} values that jointly satisfy every remaining assumption, this countermodel exercises the nested compilation machinery directly, and it confirms that \code{cP_corresponds_to_sP} is independent of the other eight assumptions of its proof.
