In my dissertation [Wheeler 2009] I provided three proofs, and showed that the
assumptions of each of the proofs were consistent.
This paper extends the dissertation, by showing that
each of the assumptions in each proof
are independent of the other assumptions.

Given some set of consistent assumptions Q, and another assumption P,
then P is independent of Q if both Q&cup;P and Q&cup;&not;P yield consistent assumptions, 
This is noted, for example, in [Kunen 1980, page xi].
Hesseling explains that Heyting [Heyting 1930] uses this approach to prove the independence of his
formalization of intuitionist logic, and credits Bernays with the technique in which
"what we would nowadays call models in which each time ten of the eleven
given axioms are true, the eleventh one false... [by] varying the
eleventh axiom and presenting a different model to which the new
combination of axioms applies, he proves that all the axioms are independent
of each other." [Hesseling, page 278].

The model-finding "mace4" tool, part of the prover9 suite,
can prove that assumptions are independent of the other assumptions.
Mace finds models that satisfy all the claims given.
An assumption X can be shown to be independent of the others if all
assumptions are given to mace, except that X is negated.
If mace can find a model that satisfies this set, then it is possible
that the negated assumption does not follow from the others, and therefore
the assumption is independent.

## The Quantifier Trap: Proving Axiom Independence Safely in Mace4

**By David A. Wheeler**

When formally verifying a system like Diverse Double-Compiling (DDC), we must ensure our foundational specifications are both sound and elegant. A key part of this process is establishing **axiom independence**—demonstrating that none of our starting assumptions can be derived from the others.

To prove independence using a model finder like Mace4, the strategy is straightforward in theory: you search for a model where all your other assumptions hold true, but the target assumption is false. If the tool finds such a model, you have successfully proven independence.

However, first-order logic tools have strict, subtle syntactic behaviors. If you are not extremely careful with how you phrase a negation, you will accidentally force the solver into a mathematical impossibility. This is exactly what happened in a recent DDC model draft, and it serves as an excellent case study in implicit quantifier scoping.

---

### The Attempted Negation

Consider the core DDC definition for a portable and deterministic program. The draft attempted to test its independence by prefixing it with a negation symbol (`-`) inside the `formulas(assumptions)` block:

```prolog
formulas(assumptions).

  - ( ( portable_and_deterministic(Source, Language, Input) &
        exactly_correspond(Executable1, Source, Language, Environment1) &
        exactly_correspond(Executable2, Source, Language, Environment2)) ->
   	  ( converttext(run(Executable1, Input, EnvEffects1, Environment1), Environment1, Target) =
        converttext(run(Executable2, Input, EnvEffects2, Environment2), Environment2, Target)))
    # label(define_portable_and_deterministic).

  % ... (other DDC axioms) ...

end_of_list.

```

At first glance, this looks completely reasonable. We want the rule to fail, so we put a minus sign in front of it. Unfortunately, this creates a catastrophic structural error due to how Mace4 handles unquantified variables.

---

### Why It Is Wrong: How Mace4 Interprets Free Variables

In Prover9 and Mace4, when you use capitalized terms (like `Source`, `Language`, `Input`) without an explicit quantifier, the tool treats them as **free variables**. To process the file, the engine must bind these variables. It does this by automatically wrapping the entire expression in an implicit universal quantifier (`all`).

Crucially, because the negation symbol (`-`) was written *inside* the statement, Mace4 applied the implicit `all` **outside** the negation.

Mathematically, Mace4 interpreted your assumption as:


$$\forall \text{ Vars } \neg (\text{Premise} \rightarrow \text{Conclusion})$$

For an implication ($\rightarrow$) to be false, the premise must be true **and** the conclusion must be false. Because of the universal quantifier, you didn't ask Mace4 to find a single, specific loophole. Instead, you commanded it to construct a universe where:

1. **Every single object** in existence must satisfy the premise (meaning *everything* must be portable, deterministic, and exactly corresponding).
2. **Every single compiled output** in existence must simultaneously *fail* to be equal to every other output.

This defines a completely absurd, hyper-restrictive mathematical universe. The moment Mace4 evaluated your other DDC axioms against this sweeping rule, it hit an immediate contradiction. The tool didn't fail because your logic was flawed; it failed because you asked it to build an impossible world.

---

### What Was Meant Instead

To prove independence, you do not want to say that the rule fails *everywhere for everything*. You want to prove that there is **at least one loophole**—a single, lonely counterexample where the prerequisites are met, but the conclusion fails to hold.

Mathematically, a true counterexample requires an **existential quantifier** ($\exists$) wrapped around the negation:


$$\exists \text{ Vars } (\text{Premise} \land \neg \text{Conclusion})$$

In raw Mace4 syntax, you would have to manually write out every single variable to force this behavior:

```prolog
% Manual existential counterexample (highly error-prone)
exists Source Language Input Executable1 Environment1 Executable2 Environment2 EnvEffects1 EnvEffects2 Target (
    portable_and_deterministic(Source, Language, Input) & ... &
    - ( converttext(...) = converttext(...) )
).

```

Writing this out manually is tedious and highly susceptible to typos or missed variables, which brings us to the correct, engineering-focused solution.

---

### The Elegant Solution: Using the `formulas(goals)` Block

The safest and most effective way to handle independence proofs in the LADR suite is to let the tool manage the quantifiers for you. Instead of manually negating the rule and stuffing it into the assumptions list, you should leave the statement **completely positive** and place it inside a **`formulas(goals)`** block.

```prolog
formulas(assumptions).
    % ... Keep all your other DDC axioms here unmodified ...
end_of_list.

formulas(goals).
  % Keep the target rule POSITIVE here. Do not use '-'
  ( ( portable_and_deterministic(Source, Language, Input) &
      exactly_correspond(Executable1, Source, Language, Environment1) &
      exactly_correspond(Executable2, Source, Language, Environment2)) ->
   	( converttext(run(Executable1, Input, EnvEffects1, Environment1), Environment1, Target) =
      converttext(run(Executable2, Input, EnvEffects2, Environment2), Environment2, Target)))
   # label(define_portable_and_deterministic).
end_of_list.

```

### Why This Is Better

When Prover9 or Mace4 sees a statement in the `goals` block, it utilizes a built-in automated mechanism:

1. It automatically treats the free variables as universally quantified for the goal ($\forall \text{ Vars } (\text{Rule})$).
2. Because the solver's job is to either prove the goal true (Prover9) or find a falsifying countermodel (Mace4), **the tool automatically negates the entire block correctly behind the scenes**.
3. It pushes the negation *inside* the universal quantifier, mathematically transforming it into a clean, localized existential statement: $\exists \text{ Vars } \neg (\text{Rule})$.

By utilizing the `goals` block, you eliminate the risk of quantifier mismatch. Mace4 will search for a model that satisfies all your assumptions while breaking the goal in exactly *one* valid place. If it finds that model, your independence proof is secure, your syntax remains clean, and you avoid the subtle scoping traps that completely upend formal analysis.

NOTES:

Proving independence in Mace4 for a complex specification like Diverse Double-Compiling (DDC) is notoriously tricky. The primary bottleneck here isn't the number of predicates, but the **combinatorial explosion caused by your functions**.

Because Mace4 searches for *finite total models*, every function you define must have an explicit output mapped for every single combination of inputs. Your `compile` function takes 5 arguments; in a domain of just size 3, that requires mapping $3^5 = 243$ combinations. When nested inside `extract` and `converttext`, the search space balloons instantly.

Here are the most promising strategies to get Mace4 to find a model without changing the underlying logic of your DDC proof.

---

### 1. Flatten the Nested Functions into Predicates

The single most effective way to speed up Mace4 is to replace deeply nested functions with relation predicates.

* **The Problem:** `extract(converttext(run(...)))` forces Mace4 to evaluate an astronomical number of function tables.
* **The Fix:** Instead of a 5-argument function for `compile`, use a 6-argument predicate: `is_compiled_to(Source, Compiler, EnvEffects, RunOn, Target, Output)`.
* **Why it works:** Mace4 evaluates predicates much faster than functions because it doesn't have to guarantee "totality" (an output for every single junk combination of inputs) for predicates.

### 2. Introduce Intermediate Constants (Manual Skolemization)

If you want to keep the functions, break up the deeply nested definitions by introducing named intermediate constants in your assumptions.

* For example, instead of letting Mace4 evaluate `compile(sA, stage1, e2effects, e2, eArun)` dynamically, explicitly define what the inner parts look like using dummy constants (e.g., `const_retarget1`, `const_run1`).
* This gives Mace4 fixed reference points ("anchors") in the domain, drastically cutting down the isomorphic models it has to test.

### 3. Aggressive Axiom Trimming (Slicing)

When proving the independence of a specific axiom, you often do not need the *entire* DDC specification.

* Identify which predicates and functions are actually involved in the axiom you are trying to negate.
* Temporarily comment out any definitions, function orders, or assumptions that do not structurally connect to that target axiom. If `converttext` isn't strictly necessary to falsify your target axiom, drop it for this run.

### 4. Leverage Mace4-Specific Parameter Tuning

You’ve already started tweaking `selection_measure`, but you can guide the search more aggressively:

* **`assign(isomorphism_testing, 1).`** Ensure this is on. It prevents Mace4 from wasting time looking at models that are just permutations of previous ones.
* **`assign(max_size, 4).`** Force a strict ceiling. If DDC has a countermodel, it’s likely either very small (size 2 or 3) or too large for Mace4 anyway. Don't let it waste time grid-searching size 5+.
* **Explicit Element Naming:** Use `assign(domain_size, 3).` and explicitly name domain elements (`0, 1, 2`) in a temporary weight/order list to force Mace4 to assign your constants (`sP`, `lsP`, `eA`) to distinct elements right out of the gate.

### 5. Consider an SMT Solver (The "Plan B")

If Mace4 continues to spin its wheels, it’s likely because the model requires an infinite domain or an algebraic structure Mace4 isn't optimized for.

* **Z3 / Alloy:** Translating these first-order logic formulas into an SMT solver like **Z3** (using Python) or using **Alloy** can be much faster. SMT solvers handle uninterpreted functions naturally without requiring the finite totality tables that bog down Mace4.
